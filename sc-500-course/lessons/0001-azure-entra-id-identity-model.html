The Azure Identity Model

Entra ID, Tenants & Subscriptions

📚 Track 1: Identity, Access & Governance ⏱ ~35 min 🗓 Lesson 0001
Lesson goal: Build the mental model for where identity lives in Azure — the relationship between Entra ID tenants, Azure subscriptions, and why they are the security boundary you'll be defending in the SC-500 exam.

1. What is Microsoft Entra ID?

Microsoft Entra ID (formerly Azure Active Directory) is Microsoft's cloud-based identity and access management service. It is not a domain controller. It is not Active Directory on-prem. It is a multi-tenant, cloud-native directory that answers one question:

The core question Entra ID answers:
"Is this principal (user, app, service) who they claim to be, and what are they allowed to do?"

Everything in Azure security starts here. Conditional Access, PIM, managed identities, app registrations — they are all capabilities layered on top of this identity foundation. [Entra ID fundamentals — Microsoft Learn]

2. The Tenant: Your Security Boundary

When an organisation signs up for a Microsoft cloud service, a tenant is created. Think of it as a dedicated, isolated instance of Entra ID.

  • One organisation typically has one tenant (sometimes more for isolation).
  • A tenant owns one or more Azure subscriptions.
  • The tenant is the trust boundary: users inside it are trusted by default; external users are guests.
  • Every tenant has a globally unique Tenant ID (a GUID) and a default domain like contoso.onmicrosoft.com.
Common misconception: "The subscription is the security boundary." It is not. The tenant is where identity lives. Subscriptions are billing/resource containers — they trust their parent tenant for authentication.

3. Subscriptions & Management Groups

An Azure subscription is a logical container for Azure resources (VMs, storage, databases). It has two key security properties:

  1. It is associated with exactly one Entra ID tenant (can be transferred).
  2. It is a billing boundary and an RBAC scope — permissions granted at subscription scope apply to all resources inside it.

Management Groups sit above subscriptions. They let you apply Azure Policy and RBAC across many subscriptions at once — critical for governance at enterprise scale.

4. Key Identity Principals

Not all identities are humans. Entra ID recognises three principal types you'll secure in the exam:

PrincipalWhat it isSC-500 relevance
User A human account in the tenant PIM, Conditional Access, MFA, SSPR
Service Principal The identity of an application registered in Entra ID App registrations, OAuth consent, permission grants
Managed Identity An automatically managed service principal for Azure resources (no password needed) Eliminates secrets in code; secure access to Key Vault, storage, etc.
Security mindset link: From your threat modelling work, think of each principal type as a trust boundary crossing point. Every time a principal gains access to a resource, that's a potential lateral movement vector. Managed identities reduce this risk by removing long-lived credentials.

5. RBAC: Who Can Do What, Where

Azure Role-Based Access Control (RBAC) controls what actions a principal can perform on Azure resources. It has three components:

  • Security Principal: user, group, service principal, or managed identity
  • Role Definition: a set of allowed operations (e.g., Reader, Contributor, Owner)
  • Scope: management group → subscription → resource group → resource

RBAC assignments are additive — a principal gets the union of all roles assigned to them across all scopes. There are no explicit deny assignments in basic RBAC (though Azure denies outside assignment scope).

Lesson 2: Boundaries, Assumptions, and Sufficient Verification

In this lesson: you will learn how to decide where trust changes, how to turn assumptions into productive review questions, how to judge whether a control is sufficiently verified, and how to review unfamiliar platforms without bluffing.

Your mission is not to produce a beautiful threat model diagram. Your mission is to lead security reviews with judgment. That means finding the places where the system is forced to trust something, then asking whether that trust is deserved.

1. How to Determine a Trust Boundary

A trust boundary is not a box on a diagram. It is a change in what the system can safely assume.

Ask who controls it

If a different party controls the component, runtime, storage, or network path, trust probably changed.

Ask what must be verified

If the receiver must verify identity, authorization, integrity, freshness, or input quality, there is likely a boundary.

Ask what a hostile sender could change

If a malicious sender could alter meaning, state, or privilege, you have found a boundary worth modeling.

For your OAuth example, the browser is not merely a client. It is a trust boundary because the application must treat the browser runtime and storage as attacker-influenced. That single judgment changes the whole control discussion around token storage, XSS, revocation, and replay.

Useful sentence: “A trust boundary exists wherever the receiver must stop trusting and start validating.”

2. How to Think About Assumptions

Assumptions become dangerous when they stay implicit. Good reviewers force them into plain language.

  1. Write the assumption as a complete sentence: We assume the revocation signal reaches the resource server immediately.
  2. Flip it: What if it does not?
  3. Name the impact: stale access, privilege persistence, incident containment failure.
  4. Ask for the control and evidence.
Weak assumption: “Tokens are stored securely.”
Stronger assumption: “Refresh tokens stored in the browser cannot be read by JavaScript, malicious extensions, backup tooling, or other apps with normal platform permissions.”

3. When Is Verification Sufficient?

Sufficient verification is not “someone said the control exists.” It is the point where a reasonable reviewer can trust the control for the risk being discussed.

Design level

You understand what the control is intended to stop.

Implementation level

You can show where the control is implemented and configured.

Runtime level

You have evidence it works under normal and failure conditions.

For a high-risk control, design evidence alone is not enough. For example, “we blacklist revoked JWTs” is only a design claim until you see:

  • where revocation state is written,
  • how the resource server reads or syncs it,
  • how long propagation takes,
  • what happens if the channel is stale or unavailable,
  • and whether negative tests prove revoked tokens are denied.
Heuristic: verification is sufficient when you can explain the control’s threat, mechanism, failure mode, and evidence without hand-waving.

4. Suggested Controls Need Verification Too

When you recommend a control, you should also recommend how to verify it. Otherwise you leave the team with a design noun rather than an actionable review outcome.

Example:

  • Suggested control: short-lived JWT access tokens.
  • Why: reduces stolen-token usefulness.
  • Verification: inspect token lifetime config, confirm resource server enforces exp, and test with expired tokens.

5. Reviewing Unfamiliar Areas

You do not need instant expertise to review a new platform. You need a disciplined way to borrow the platform’s own standards.

  1. Admit the platform shift explicitly: web assumptions do not transfer cleanly to mobile.
  2. Find the platform standard that defines “good enough.” For web, use OWASP ASVS. For mobile, use OWASP MASVS and OWASP MASTG.
  3. Translate your boundary questions, not your favorite controls.
  4. Ask what attacker capabilities are normal on this platform.
  5. Use the testing guide to generate concrete verification questions.
Common mistake: a web-focused reviewer asks only about API auth and misses mobile-specific boundaries like deep links, WebViews, screenshots, device backups, rooted devices, and local inter-process communication.

6. A Portable Question Set

When you are unfamiliar with a domain, ask these in order:
  1. What are the main assets?
  2. Where does trust change?
  3. What assumptions are required for this design to be safe?
  4. Which controls enforce those assumptions?
  5. What evidence proves those controls work here?
  6. Which platform standard should I use to fill my knowledge gaps?

Recommended Reading

Primary sources: OWASP ASVS for verification thinking, and OWASP Mobile Application Security with MASVS and MASTG for reviewing mobile systems outside a web-specialist comfort zone.

Practice now:

Pick one existing review you did recently. Write down:

  1. one trust boundary you initially under-specified,
  2. one assumption that should have been explicit,
  3. one control you accepted too easily,
  4. and one piece of runtime evidence you should have demanded.

This is how the mindset gets sharper.

Reference companion: use the Security Review Checklist during live reviews, and keep the Glossary nearby for consistent terms.

Ask follow-up questions. The fastest way to build reviewer judgment is to take a real boundary or control from your work and stress-test it conversationally.

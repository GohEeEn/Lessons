Lesson 4: Design-Phase Threat Modeling Inputs and Outputs

In this lesson: you will learn exactly what input artifacts you need before code exists, what output artifacts you must produce, and how to make design-phase threat modeling testable by downstream teams.

Goal: produce a threat model that is useful now (for architecture decisions) and useful later (for implementation verification and runtime assurance). In design phase, you cannot prove control effectiveness yet, but you can produce a high-quality security contract.

1. Required Inputs for Design-Phase Threat Modeling

You do not need final code. You do need enough structure to reason about trust and failure.

Input Minimum required content Why it matters
System scope In-scope components, out-of-scope components, external dependencies Prevents threat model drift and vague coverage
Business context Primary use cases, critical business outcomes, unacceptable failures Anchors prioritization on business harm
Asset inventory Data classes, secrets, privileged operations, integrity-sensitive states Defines what must be protected
Architecture draft Component diagram and data flow diagram with trust boundaries Enables attack-path reasoning before implementation
Identity model Actors, authn/authz model, token/session strategy, privilege boundaries Most critical failures often involve identity misuse
Assumptions list Explicit "we assume" statements Turns hidden dependencies into review targets
Constraints Regulatory, platform, latency, cost, operational constraints Controls must be feasible and sustainable
Anti-pattern: starting threat modeling with only a feature list and no trust boundaries. That produces generic findings and weak mitigations.

2. Mandatory Outputs from Design-Phase Threat Modeling

Your output should function as a security contract that downstream teams can verify.

A. Threat register

Each threat should include: boundary, attacker capability, assumption violated, impact, and priority.

B. Control decisions

For each high-risk threat, define one or more controls and rationale.

C. Verification requirements

For each control, define what evidence is needed at build and runtime stages.

D. Security backlog

Create actionable tickets with owners, acceptance criteria, and release gates.

3. Output Quality Bar: “Testable Security Statements”

Replace vague statements with verifiable requirements.

Weak output Strong output Later verification hook
Use secure auth Resource server must reject tokens with invalid issuer, audience, expiry, signature, and scope. Negative integration tests and production auth failure telemetry
Protect tokens Refresh tokens must not be readable by browser scripts and must be rotated with reuse detection. Code/config review plus replay test case
Enable revocation Revocation propagation delay must remain below defined threshold, and stale revocation state must fail closed for privileged operations. Operational SLO and chaos/failure-path validation

4. Ownership Model Across Phases

Design phase produces the contract. Build phase proves implementation alignment. Runtime phase proves operational effectiveness.
Phase Primary owner Expected evidence
Design Product architecture plus security reviewer Threat register, control decisions, testable requirements, release gates
Implementation Engineering plus AppSec and security tooling Tests, static checks, code review evidence, secure configuration checks
Runtime Operations plus detection engineering plus pentest Monitoring, alerting, abuse-case simulation, pentest results, incident-readiness checks

5. A Practical Workflow (Design Only)

  1. Collect input artifacts and mark known gaps explicitly.
  2. Draw data flows and trust boundaries.
  3. Write top assumptions as complete statements.
  4. Generate threats by boundary crossing and assumption failure.
  5. Select control strategies and document tradeoffs.
  6. Translate controls into testable requirements.
  7. Create downstream verification tasks and release gates.
Completion criteria in design phase:
  • You can explain why each major control exists.
  • Each control has a future verification path.
  • Unknowns are explicit and tracked, not hidden.
  • No critical threat is left with "we will figure it out later."

Security Review Checklist

Trust Boundaries, Assumptions, and Control Verification

Use this during live reviews. Start at the boundary, force assumptions into explicit statements, then demand evidence that each control actually works in the real system rather than only in the design slide.

1. Find the Trust Boundary

Definition

A trust boundary exists wherever the system must stop assuming the other side is safe, correct, authenticated, or under the same controls.

Boundary Signals

Different owner, different runtime, different privilege, different network zone, different storage, or different identity context.

Fast Test

If the receiver must validate identity, integrity, authorization, freshness, or input quality, you are probably crossing a trust boundary.

Questions to ask:
  • Who controls each side of this interaction?
  • What data, identity, or command crosses here?
  • What can the sender influence?
  • What does the receiver have to verify before trusting it?
  • What changes if the sender is malicious, buggy, or stale?

2. Turn Assumptions into Review Questions

Assumptions are hidden design dependencies. Write them as explicit statements, then try to break them.

Assumption Pattern Review Question Threat if False
We assume the client is well-behaved. What happens if the client is modified, replaying, or malicious? Bypass, replay, forged requests
We assume tokens remain confidential on the device. How are tokens protected from XSS, extension abuse, logs, backup, and debug tooling? Session theft, persistent access
We assume revocation is immediate. What exact mechanism propagates revocation, and what is the worst-case delay? Revoked tokens remain valid
We assume the identity provider is the source of truth. How does the resource server authenticate metadata, JWKS, introspection, or denylist state? Forged trust, stale validation
We assume the platform storage is secure. Secure against whom: other apps, local malware, backups, rooted devices, developers, browser scripts? False confidence, token leakage
Review posture: never leave an assumption as a sentence fragment like “tokens are stored securely.” Force the team to define actor, mechanism, scope, and failure mode.

3. Decide Whether Control Verification Is Sufficient

A control is sufficiently verified when you have enough evidence to trust it in the actual operating environment, not just the architecture diagram.

Design Evidence

Requirements, architecture, protocol choice, configuration intent.

Implementation Evidence

Code path, framework settings, exact validation logic, storage and transport details.

Runtime Evidence

Logs, telemetry, negative tests, operational checks, incident behavior, failure modes.

Sufficient verification heuristic:
  • You can state what the control is meant to stop.
  • You can point to where it is implemented.
  • You have tested both the expected path and a failure path.
  • You know how it fails under degradation or outage.
  • You know who would detect that failure.
Control Type Minimum Verification Stronger Verification
Token validation Show issuer, audience, expiry, signature, and scope checks exist. Run negative tests with wrong issuer, expired token, wrong audience, revoked token.
Refresh-token rotation Show rotation logic and reuse detection rules. Demonstrate stolen-old-token replay causes invalidation and alerting.
Immediate revocation Show the revocation channel and propagation design. Measure revocation latency and fail behavior during auth-server outage.
Secure storage Identify exact storage mechanism and access paths. Show what a hostile script, extension, backup, debugger, or rooted device can and cannot access.
XSS mitigation Show output encoding and CSP design. Test realistic injection paths and verify token exfiltration is blocked.

4. Ask Better Questions in Unfamiliar Domains

When you review outside your home territory, do not pretend to be a domain expert. Borrow the domain’s verification standard and ask boundary-first questions.

Portable reviewer script:
  1. What are the assets and trust boundaries in this platform?
  2. Which standard defines “good enough” here?
  3. Which controls are platform-native versus app-implemented?
  4. What attacker capabilities are normal for this platform?
  5. Which test guide gives me concrete checks?
If you know web... Ask in mobile... Source
Where are session tokens stored? Which platform storage is used: Keychain, Keystore, encrypted shared prefs, custom store? OWASP MASVS
How is input validated? What deep-link, IPC, URL scheme, or WebView inputs exist? OWASP MASTG
Do we rely on TLS? Is TLS pinning used, and what is the fallback/failure behavior? OWASP MASTG
Can scripts steal secrets? Can WebViews, logs, screenshots, backups, rooted devices, or other apps reach secrets? OWASP MAS Project

5. Recommended Review Flow

  1. Draw the data flow and mark trust boundaries.
  2. Write the top assumptions as explicit sentences.
  3. Convert each assumption into a hostile question.
  4. Map one or more controls to each high-risk boundary crossing.
  5. Demand design, implementation, and runtime evidence.
  6. Escalate unresolved unknowns to the platform standard: OWASP ASVS for web, OWASP MASVS and OWASP MASTG for mobile.
